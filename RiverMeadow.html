Validate the deployment

  1. Login to the AWS burner account and switch region to us-east-2
  2. Select EC2 → and validate if 4 EC2 instances with the names similar to the screenshot below are deployed and instance in running state
  3. Search for peering connections on the AWS console, select peering connection and make sure status as Active
  4. Check the us-east-1 and us-east-2 route tables named “Source-RouteTable” and “Target-RiverMadeow-RouteTable” to make sure the routes 172.31.0.0/20 and 172.41.0.0/20 are added
  5. Go to IAM → Select Roles → search for RiverMeadow
  6. Make sure the IAM roles are created and available

Configuring RiverMeadow SaaS platform

  1. Log into the RiverMeadow link
  2. Select Manage → Migration Appliances
  3. Currently there will be no migration appliance available. Select “Add new migration appliance” on left side and select cloud type as → AWS
  4. Go to the AWS Console, → IAM → Roles → Search RiverMeadow → Click on the role → Copy the ARN
  5. On the RiverMeadow console, Provide any name → Paste the ARN for the role copied from the RiverMeadow console and test connection. Keep the “Setup Migration Appliance for this target cloud” option selected. Click on add migration appliance (Note : There are two groups accessing the same RiverMeadow project for this Hackathon. Hence to avoid confusion, please make sure you name you appliance with your group identification. Refer to screen shot below)


  6. In the Next screen, Select Region → us-east-1 (US East (N.Virginia)) → VPC to Target-RiverMeadowVPC → Subnet to TargetRiverMeadowSubnet → Enable Public Ip option as "Use Subnet Settings" and provide an instance name of your choice as per image below and select Initiate Migration appliance checks

  7. Wait for the appliance to be deployed
  8. Wait for the “Appliance Readiness Status” to be shown as “Ready” (refresh the page if status is not updated automatically in 3-5 minutes)



IMPORTANT : DO NOT MISS THIS STEP and EXECUTE IMMEDIATELY ONCE THE SECURITY GROUP IS CREATED

As RiverMeadow migration appliance is deployed to the AWS Burner account, it automatically creates a security group with name similar to “RM-CA-vpc-xxxxxx” and with inbound rules as given below with source as 0.0.0.0/0 ( 4 rules - Time exceeded, Destination unreachable, Echo request & Echo reply). The burner account info security policy does not allow source as 0.0.0.0/0 which will lead to burner account being auto closed.

To avoid this issue, please modify the inbound rules with source as 0.0.0.0/0 and replace with your source vpc subnet (us-east-2) CIDR, which is 172.31.0.0/20 

Refer the below image for security group rules post changes

Add Source servers for migration to RM-SaaS

  1. Login to the RM portal using the link
  2. Select Manage → Source Inventory. The Source Inventory page appears.
  3. Click the Add Source button. The Add a Source form appears. Make sure the source has preinstalled agent option is selected. Use the existing move group with your group name. Click on add source (Note : There are multiple groups accessing the same project for this Hackathon. Hence to avoid confusion, please make sure you move the source servers you add to the correct move group with your group name. If none exists, please create new using your group name)
  4. Add all the four source servers part of the migration. 
  5. Once added the source servers, you can begin the migration process from this page by selecting your source(s) and clicking the "Inspect Sources"  option first. This step will capture the metadata details of the server.

Migrate Source Servers to AWS

  1. Login to the RM SaaS using the link and select “Start Migration” from the “Migrate option”
  2. Select configure Migration
  3. Select Workload Migration from the dropdown and check migration readiness
  4. Wait for all the checks to be completed and status as ready. Once ready, select Create Migration Profile
  5. Specify migration details, including configuration details for your Target Instances (server clones).
  6. REVIEW AND APPLY THE BELOW SECTIONS BASED SOURCE SERVERS BASED ON THE SCENARIOS GIVEN IN HACKATHON
  7. Select the Storage RightSizing option under “Optimisation Settings” if you intent to resize the storage

  8. Choose the “Modernization Settings” if you intent to Modernize/Upgrade Operating systems. For Redhat servers or Windows servers, you can choose a OS type under Modernization settings to upgrade your operating system to latest version. For servers with SQL servers installed, below steps can be followed for SQL upgrade

  9. Choose the “Security Settings” if you intent to configure the target server to specific security groups

  10. Choose the advanced options (as applicable to your scenario) under the advanced settings



  11. Initiate pre-migration checks
  12. Once the migration is started, you can view and monitor the progress
  13. You will receive an email on your amazon email with a successful migration completion message. Also, please login to the target migrated servers using provided credential for validation
